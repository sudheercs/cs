1. What is Problem Solving in CS?

It's not just about writing code.

🧠

Critical Thinking First

Programming is merely a tool. The core of Computer Science is computational thinking—the ability to define a problem, structure a solution, and design a logical progression of steps to solve it.

⚠️

The Danger of Rushing

Jumping straight into typing code without a plan almost always results in buggy, unmaintainable, and inefficient software.

3. Step 1: Analyzing the Problem

Breaking it down

Before designing a solution, you must perfectly understand the boundaries of the problem. This involves identifying three core components:

  1. Inputs: What information is given or required?
  2. Outputs: What is the exact expected result?
  3. Constraints: What are the limitations? (e.g., memory limits, speed, specific data ranges).

Example Scenario

"Calculate the average temperature of a city over a week."

  • Inputs: 7 numeric temperature values.
  • Outputs: 1 floating-point number (the average).
  • Constraints: Cannot divide by zero. Temps can be negative.

4. Step 2: Developing an Algorithm

The architectural blueprint.

What is an Algorithm?

An algorithm is a finite, well-defined sequence of unambiguous instructions used to solve a specific problem.

Algorithms are language-independent. They represent the logic, not the syntax of Python or Java. You write algorithms before you write code.

Characteristics of a Good Algorithm

  • Finiteness: Must eventually terminate.
  • Definiteness: Each step must be painfully clear.
  • Effectiveness: Steps must be basic enough to actually be executed.

5. Step 3: Coding

Translating logic into machine instructions.

Implementation

This is where you choose a programming language (like Python, Java, or C++) and translate your abstract algorithm into specific syntax.

The focus here shifts from pure logic to obeying the strict grammatical rules (syntax) of the compiler/interpreter.

Good Coding Practices

  • Writing meaningful variable names.
  • Adding comments to explain why code exists.
  • Maintaining proper indentation and formatting.

6. Step 4: Testing

Verifying that the code actually solves the problem.

Why Test?

Your code might compile perfectly, but still produce the wrong mathematical answer. Testing involves feeding specific data (where the output is known) into the program to verify behavior.

Normal Data

Typical, expected inputs.
(e.g., finding average of 70, 80, 90).

Boundary Data

Extreme inputs at the edge of limits.
(e.g., maximum possible integer, or 0).

Erroneous Data

Invalid inputs to test error handling.
(e.g., calculating average of "Hello" and "World").

7. Step 5: Debugging

Identifying and removing errors from software.

Debugging is arguably the most time-consuming part of programming. When a test fails, you must hunt down the bug. Errors fall into three main categories:

1. Syntax Errors

Grammar mistakes. The compiler cannot understand your code.

Example: Missing a semicolon, misspelling `print` as `prnt`, unclosed quotes.

2. Logical Errors

The program runs, but gives the wrong answer. Hardest to find.

Example: Writing Area = length + width instead of length * width.

3. Runtime Errors

The program compiles, but crashes while running.

Example: Trying to divide a number by zero, or accessing memory that doesn't exist.

8. Problem Decomposition

Divide and Conquer.

What is it?

Decomposition (or Top-Down Design) is the process of breaking down a large, complex, and unmanageable problem into smaller, simpler, and isolated sub-problems.

Why is it important?

  • Smaller problems are much easier to conceptualize and solve.
  • Allows multiple programmers to work on different parts simultaneously.
  • Reduces duplicated code through reusable modules/functions.

11. Representing Algorithms

How do we communicate our logic before writing code?

1. Pseudocode

An artificial, informal language that helps programmers develop algorithms.

It reads like English but utilizes programming structural conventions (IF, ELSE, LOOP).

2. Flowcharts

A diagrammatic, visual representation of an algorithm.

Uses standardized geometric shapes to map out the exact flow of execution and decision making.

12. Pseudocode Basics

Rules of Pseudocode

  • There is no strict syntax. As long as it is logical and unambiguous, it is valid.
  • Capitalize keywords (INPUT, PRINT, IF, WHILE) for readability.
  • Use indentation to indicate blocks of code (inside loops or conditionals).

Common Keywords

Action Keyword
Getting data INPUT, READ, GET
Showing data PRINT, DISPLAY, OUTPUT
Math COMPUTE, CALCULATE
Decisions IF, ELSE, THEN
Looping WHILE, FOR, REPEAT UNTIL

13. Pseudocode: Example 1

Write an algorithm to find the largest of two numbers.

Algorithm Logic:

START
  INPUT Num1
  INPUT Num2
  
  IF Num1 > Num2 THEN
    PRINT "Num1 is larger"
  ELSE IF Num2 > Num1 THEN
    PRINT "Num2 is larger"
  ELSE
    PRINT "Both are equal"
  END IF
END

14. Pseudocode: Example 2

Algorithm to check if a number is Even or Odd.

Algorithm Logic:

START
  PRINT "Enter a number:"
  INPUT Num
  
  REMAINDER = Num % 2
  
  IF REMAINDER == 0 THEN
    PRINT "Number is Even"
  ELSE
    PRINT "Number is Odd"
  END IF
END

15. Introduction to Flowcharts

The visual blueprint of an algorithm.

What is a Flowchart?

A flowchart is a type of diagram that represents a workflow, process, or numeric algorithm. Step-by-step approaches to solving a task are shown using different shaped boxes, connected by arrows.

Why use them?

  • Visuals are easier for humans to understand than text.
  • They clearly expose logical flaws and infinite loops.
  • Excellent for communicating logic to non-technical stakeholders (managers/clients).

16. Standard Flowchart Symbols (Part 1)

Terminal (Oval)

Indicates the beginning or end of a program.

I/O (Parallelogram)

Used whenever data is read from the user or displayed on the screen.

17. Standard Flowchart Symbols (Part 2)

Process (Rectangle)

Used for mathematical calculations, variable assignments, and data manipulation.

Decision (Diamond)

Represents a condition (IF statement). Contains a Yes/No (True/False) branch. This is the only shape that has more than one arrow coming out of it.

18. Flowchart Rules & Best Practices

✅ Do:

  • Ensure there is exactly ONE start oval, and at least ONE stop oval.
  • Draw arrows (flowlines) strictly from top to bottom, or left to right.
  • Keep logic simple. If it's too large, use connector symbols.

❌ Don't:

  • Have flowlines cross over each other (causes confusion).
  • Have multiple arrows exiting a Processing rectangle (only Diamonds split paths).
  • Use code-specific syntax inside the boxes (use plain English formulas).

20. Advanced Concept: Loops in Flowcharts

How do we Loop?

A loop is represented simply by drawing a flowline (arrow) pointing backwards (upwards) to a previous step in the flowchart.

The loop must ALWAYS be controlled by a decision diamond to prevent an infinite loop!

21. Pseudocode vs Flowcharts

Flowcharts Pseudocode
✅ Easy to trace logic visually at a glance. ✅ Very fast to type out.
✅ Excellent for absolute beginners. ✅ Closer to actual programming code.
❌ Extremely time consuming to draw/modify. ❌ Harder to visualize loops and complex deeply-nested IF statements.
❌ Becomes a chaotic web of arrows for large, complex problems. ✅ Highly scalable for complex algorithms.

22. Bridging the Gap: Structure to Code

Algorithm

IF Age >= 18 THEN
  PRINT "Eligible"
END IF

Python Code

if age >= 18:
  print("Eligible")

C++ / Java Code

if (age >= 18) {
  cout << "Eligible";
}

Notice how the Core Logic (the problem solving aspect) remains completely untouched across all instances. Only the grammar (syntax) changes.

23. Summary: The Golden Rule

"Think before you type."

Never start coding a complex problem immediately.

1. Analyze inputs and outputs.
2. Decompose into smaller sub-problems.
3. Draft your logic using Pseudocode or Flowcharts.
4. Then execute the code, test it rigorously, and debug your errors.
